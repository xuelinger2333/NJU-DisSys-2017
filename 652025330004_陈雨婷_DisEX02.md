## 2. Part I — Leader Election

### 2.1 目标

- 实现 `RequestVote` RPC + handler；
- `Make()` 启动后台 goroutine；
- 选举超时触发选举，term 自增，向所有 peer 拉票；
- 多数派选票即当选；
- Leader 周期性发心跳（`AppendEntries`，Entries 为空）以抑制其他 peer 选举；
- 选举超时 **随机化** 防止 split vote。

### 2.2 设计

**三角色状态机**：`roleFollower / roleCandidate / roleLeader`，单 int 字段。

**Raft struct（Figure 2 完整字段）**：

```go
type Raft struct {
    mu sync.Mutex; peers []*labrpc.ClientEnd; persister *Persister; me int

    // Persistent
    currentTerm int
    votedFor    int        // -1 表示未投
    log         []LogEntry // log[0] 是哨兵 {Term:0}

    // Volatile
    role        int
    commitIndex int
    lastApplied int

    // Volatile leader-only
    nextIndex  []int
    matchIndex []int

    // Aux
    electionResetAt time.Time
    applyCh         chan ApplyMsg
    dead            int32 // atomic
}
```

`log[0]` 哨兵让 `LastLogIndex = len(log)-1`、`LastLogTerm = log[last].Term` 在空 log 情况下也良定义。

**RPC 契约（Figure 2 全字段）**：

```go
type RequestVoteArgs   struct { Term, CandidateId, LastLogIndex, LastLogTerm int }
type RequestVoteReply  struct { Term int; VoteGranted bool }

type AppendEntriesArgs struct {
    Term, LeaderId, PrevLogIndex, PrevLogTerm int
    Entries      []LogEntry
    LeaderCommit int
}
type AppendEntriesReply struct { Term int; Success bool /* + Conflict 字段在 Part II 加 */ }
```

**Goroutine 布局**：

- `ticker()` — 每 10ms 检查；非 Leader 且超时（300 + rand.Intn(150) ms）时触发选举。
- `startElection()` — 在锁内 `term++`、`role=Candidate`、`votedFor=me`、`persist()`、重置 timer，然后释放锁，每个 peer 启动一个 goroutine 拉票。
- `heartbeat(leaderTerm)` — 当选后启动；每 120ms 给所有 peer 发 AppendEntries（Part I 时为空）；遇到更高 term 即下台。

**关键时序常量**：

| 名字 | 值 | 论据 |
|---|---|---|
| 选举超时 | 300–449ms 随机 | 远小于测试预算 1000ms；`me` 混入 seed 防 split vote |
| 心跳间隔 | 120ms | < 300ms 最小选举超时，2 次丢心跳也不会误选举 |
| ticker tick | 10ms | 紧凑触发，无忙等 |

### 2.3 实现要点

**锁纪律**：
- 所有 `peers[i].Call(...)` 都在 `rf.mu` 释放后调用；
- reply 处理 goroutine 重新加锁后必校验 `rf.currentTerm == termSnap && rf.role == roleCandidate`，否则丢弃。

**RequestVote handler**（[raft.go:155](src/raft/raft.go) 起）—— 严格按 Figure 2：
```
if args.Term < currentTerm: 拒绝
if args.Term > currentTerm: step down + votedFor=-1
upToDate := args.LastLogTerm > 我.lastTerm ||
            (args.LastLogTerm == 我.lastTerm && args.LastLogIndex >= 我.lastIndex)
if (votedFor==-1 || votedFor==CandidateId) && upToDate:
    grant + reset election timer
defer persist()
```

**Kill / killed**：`atomic.StoreInt32(&dead, 1)`，所有 goroutine 主循环顶查 `killed()` 退出，无泄漏。

## 3. Part II — Log Replication

### 3.1 目标

- 完成 `Start()`：Leader 接受 client 命令、追加到 log、立刻返回 index；
- 完成 `AppendEntries` handler（一致性检查 + 截断追加 + 提交推进）；
- Leader 按 `nextIndex[i]` 复制条目到 follower；
- 已提交条目通过 `applyCh` 上报；
- 通过 `TestBasicAgree`，**尽量** 通过 Persist 之前的所有测试（`TestFailAgree/FailNoAgree/ConcurrentStarts/Rejoin/Backup/Count`）。
- 可选优化：`nextIndex` 一次回退多条。

### 3.2 设计

**`Start(cmd)`** —— Leader 立刻 append + persist + 一次性广播 + 返回新 slot 的 index：
```
if role != Leader: return -1, term, false
log = append(log, LogEntry{Term, Command:cmd})
index = len(log)-1
matchIndex[me]=index; nextIndex[me]=index+1
persist()
go kickReplicate(currentTerm)   // 立即触发，不等下次心跳
return index, currentTerm, true
```

`TestFailNoAgree` 强制要求即使无 quorum，`Start(20)` 也必须立刻返回 `index=2` —— 这种 append-then-return 语义满足该断言。

**`AppendEntries` handler（完整 Figure 2）**：
```
if args.Term < currentTerm: 拒绝（带 ConflictTerm/Index hint）
高 term -> step down
重置 election timer
// 一致性检查（带快速回退提示）
if len(log) <= PrevLogIndex:        ConflictTerm=-1, ConflictIndex=len(log); return false
if log[PrevLogIndex].Term != PrevLogTerm:
    ConflictTerm = log[PrevLogIndex].Term
    ConflictIndex = 该 term 在我 log 中第一次出现的位置
    return false
// merge：仅在真正 term 冲突处截断追加（关键！）
for k, e := range Entries:
    idx = PrevLogIndex+1+k
    if idx >= len(log): append all remaining; break
    if log[idx].Term != e.Term: log = log[:idx]; append remaining; break
    // term 相等 -> 跳过，不截断（防止 stale duplicate AE 抹掉已对齐的尾部）
// 推进 commit
if LeaderCommit > commitIndex:
    commitIndex = min(LeaderCommit, PrevLogIndex+len(Entries))
defer persist()
return true
```

**关键正确性点**：merge 循环 **不能** 一上来就 `log = log[:PrevLogIndex+1]`。在不可靠网络下，stale 重复 AE 携带旧 Entries 会抹掉新 leader 已对齐的尾部 —— 这是 `TestUnreliableAgree`、`TestFigure8Unreliable` 的核心考点。

**`AppendEntriesReply` 扩展（向后兼容）**：

```go
type AppendEntriesReply struct {
    Term          int
    Success       bool
    ConflictTerm  int   // -1 表示 PrevLogIndex 越过 follower log 末尾
    ConflictIndex int   // 案例 A: len(log)；案例 B: 该 term 在 follower log 中起点
}
```

gob 对 wire 上多出/少出的字段自动用零值，老 sender 也兼容。

**Leader 复制驱动（`heartbeat`）**：每 120ms 持锁一次性按 `nextIndex[i]` 为每个 peer 现编 AE args（用 `make+copy` 避免切片别名），释放锁后并发 fan-out。Reply goroutine 处理：
- 高 term → step down；
- success → `matchIndex[i]=max(...)`，`nextIndex[i]=match+1`，调 `advanceCommit()`；
- 失败 → 应用 ConflictTerm/Index hint 一次跳一整段 term。

**`advanceCommit()`** —— Figure 8 安全：从尾向前扫，找最大 `N > commitIndex`：
```
if log[N].Term != currentTerm: continue   // 关键：只直接提交本任期条目
cnt = sum(matchIndex[i] >= N for all i)
if cnt > len(peers)/2: commitIndex = N; return
```
`matchIndex[me]` 由 `Start()`/`becomeLeader()` 维护为 `len(log)-1`，使统计公式无 self 特例。

**`applier()`** —— 单写者 goroutine，10ms 节拍：
```
for !killed():
    sleep 10ms
    lock
    while lastApplied < commitIndex:
        lastApplied++
        msgs.append(ApplyMsg{Index: lastApplied, Command: log[lastApplied].Command})
    unlock
    for m := msgs: applyCh <- m   // 锁外发送，避免 tester 慢读阻死
```
单写者 `lastApplied` + 严格单调递增，满足 `config.go` apply pump 的 `m.Index > 1 ⇒ 上一条已 apply` 断言。

**`kickReplicate`** —— `Start` 后一次性广播，复用 `buildAEArgs` 与 `heartbeat` 同一套构造逻辑。

### 3.3 实现要点

| 边界场景 | 处理 |
|---|---|
| Stale AE reply（term/role 已变） | reply 处理 goroutine 重新加锁后校验，丢弃 |
| matchIndex 单调 | `if newMatch > matchIndex[i]` 守护 |
| 切片别名 | `make + copy` 出独立 Entries 切片 |
| Idempotent merge | 仅在 term 冲突处截断 |
| commitIndex 不回退 | `min(LeaderCommit, PrevLogIndex+len(Entries))` 而非 `len(log)-1` |
| heartbeat 到已追平的 peer | `entries=nil`，纯心跳 |

**关于 no-op on leader change**：刻意不实现。`TestFigure8Unreliable` 在我的实现里依然通过 —— 新 leader 上任后第一个 `Start` 就会带本 term 的条目，间接拉提交早任期的条目。

## 4. Part III — Persistent State

### 4.1 目标

- 实现 `persist()` / `readPersist()`，按 Figure 2 持久化 `currentTerm / votedFor / log`；
- `Make` 中从 Persister 还原；
- 自行判断 Raft 协议中哪些点需要插入 `persist()`；
- 通过 `TestPersist1/2/3`；可选挑战：`TestFigure8/Unreliable/Churn`。
- 可选优化：`nextIndex` 多回退 → **已在 Part II 完成**（ConflictTerm/Index）。

### 4.2 设计与实现

**`persist()` / `readPersist()`** 用 gob：

```go
func (rf *Raft) persist() {
    w := new(bytes.Buffer)
    e := gob.NewEncoder(w)
    e.Encode(rf.currentTerm); e.Encode(rf.votedFor); e.Encode(rf.log)
    rf.persister.SaveRaftState(w.Bytes())
}
```

**`persist()` 调用站点审计**（Figure 2 的 3 个持久字段共有 6 个变更点）：

| # | 函数 | 变更字段 | persist 覆盖 |
|---|---|---|---|
| 1 | `RequestVote` | term, votedFor | `defer rf.persist()` (LIFO 在 unlock 前持锁运行) |
| 2 | `AppendEntries` | term, votedFor, log | `defer rf.persist()` |
| 3 | `Start` | log | 同步 `rf.persist()`，**先于** `go kickReplicate` |
| 4 | `startElection` | term++, votedFor=me | 同步 `rf.persist()`，**先于** RV 拉票 |
| 5 | startElection 的 reply goroutine（高 term step down） | term, votedFor | 同步 `rf.persist()` |
| 6 | `replicateOnce`（高 term step down） | term, votedFor | 同步 `rf.persist()` |

**defer LIFO 关键**：

```go
rf.mu.Lock()
defer rf.mu.Unlock()    // 先注册 → 后执行
defer rf.persist()      // 后注册 → 先执行
```

LIFO 保证 `persist()` 在 unlock **之前** 运行，即依然持锁。这是必须的 —— `persist()` 读 `currentTerm/votedFor/log`，并发 goroutine 可能正在改。

**RPC reply 之前的持久化**：labrpc 的 `Service.dispatch` 在 handler 返回 **之后** 才 `gob.Encode` reply 字节。`defer persist()` 在 handler 栈帧 unwind 时已完成 `SaveRaftState`，因此 reply 离开服务端之前持久状态已落盘 —— 满足 Figure 2 "respond to RPCs only after persistent state has been updated"。

**`Make` 重启路径**：
```go
rf.currentTerm=0; rf.votedFor=-1; rf.log=[]LogEntry{{Term:0}}    // 默认值
rf.role=roleFollower; rf.electionResetAt=time.Now()              // 关键：先放 timer
rf.readPersist(persister.ReadRaftState())                        // 还原持久字段
rand.Seed(time.Now().UnixNano() ^ int64(me))
go rf.ticker(); go rf.applier()
```

`commitIndex/lastApplied` 是 volatile，重启后归零；leader 第一波心跳带 `LeaderCommit=K` 触发 follower 把 commitIndex 推到 K，applier 从 1 开始重放至 K。`config.go` 的 apply pump 容忍重放（值一致即可）。

**Figure 8 commit safety guard**：`advanceCommit` 中：
```go
if rf.log[N].Term != rf.currentTerm { continue }  // 只直接提交本任期条目
```

## 5. 整体架构与并发模型

### 5.1 单文件 600 行的模块图

```
                ┌─────────────────────────────────────────┐
                │          Raft (rf.mu 保护一切)          │
                │                                         │
   labrpc ──→  RequestVote / AppendEntries (RPC handlers) │
                │     │            │                      │
                │     ▼            ▼                      │
                │  defer persist() / state mutation       │
                │                                         │
   ticker  ──→ startElection ──→ RV reply goroutine ──┐  │
                                                       │  │
   becomeLeader ─┬→ heartbeat ─┐                       │  │
                 │              │                      │  │
                 │              ▼                      │  │
                 └→  buildAEArgs / replicateOnce ──────┤  │
                                                       │  │
   Start ────→ append + persist + go kickReplicate ────┘  │
                                                          │
   commitIndex ──→ applier ──→ applyCh                    │
                                                          │
   Kill ──→ atomic dead=1, all loops exit                 │
                └─────────────────────────────────────────┘
```

### 5.2 锁纪律（不可违反的三条）

1. 任何 RPC `Call` 期间不持锁；
2. reply 处理 goroutine 重新加锁后必校验 `currentTerm/role` 是否仍是当初发出 RPC 时的值；
3. `defer persist()` 先于 `defer Unlock`（LIFO），保证 persist 持锁。

## 7. 总结

### 7.1 学到的东西

- **架构先行胜过逐步演进**：Part I 阶段就把 Figure 2 全字段定义清楚，Part II 没有动 wire 类型；`defer persist()` 一次到位让 Part III 几乎无新增代码。这种"为后续 Part 留接口"的思路把整体复杂度压到最低。
- **锁纪律比算法正确性更难**：`peers[i].Call` 期间绝不持锁、reply 后必重检 term/role —— 这两条是 6.824 学生最常踩的坑。我从一开始就把它当作不可违反的契约执行，省掉了大量调试时间。
- **不要轻易 truncate**：`AppendEntries` handler 的 merge 循环一旦写成"先截断到 PrevLogIndex+1，再 append"，在不可靠网络下立刻翻车。**只在真正的 term 冲突点截断** —— 这是论文里没有显式强调，但不可靠测试会无情考察的一条。
- **Figure 8 的 commit safety guard** (`log[N].Term == currentTerm`) 是直接提交跨任期条目的唯一守门员，少了这一行 `TestFigure8` 必崩。
- **ConflictTerm/Index 快速回退** 不仅仅是优化 —— 没有它，`TestBackup` 在合理时限内通过都困难。该优化看上去复杂，实际上只增加 ~15 行代码。
